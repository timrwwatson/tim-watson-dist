Explicit

struct B
{
    explicit B(int) { }
    explicit B(int, int) { }
    explicit operator bool() const { return true; }
};

Code from here. The explicit keyword determines that for a given object, how it will resolve the construction to produce the object. Specifically for constructors with only one parameter. With the keyword no implicit construction can take place which means you could only construct the class above like:

B b = B(2)

and not:

B b = 2

The idea is so that you definitely call the correct construction function with the correct arguments and the compiler doesn’t attempt to guess which one you might want ie you are explicit!

Noexcept

From the docs, Every function in C++ is either non-throwing or potentially throwing. The keyword can therefore be used to tell the compiler if you expect this function to possibly throw exceptions or to never throw them. Using it to “promise” that a function won’t ever throw means everything this function can call also has this characteristic too.

Casting

Admittadly I wouldn’t consider casting to be keywords and more functions but the keywords page lists some casting as keywords.

  • static_cast
  • reinterpret_cast
  • std::bit_cast

The post here goes over the differences really well. static_cast can and should be the go-to aimed at converting between similar or related types. For things that you are certain can be converted you can use reinterpret_cast but it won’t do any checking it will just trust you know what your doing. As the post describes for many of the reasons you might want to use reinterpret_cast such as switching between two types std::bit_cast may be more appropriate. However the objects have to be the same size and easily copyable.

Public/Protected/Private

Probably some of the easier ones to understand but if I was going to go over the keywords it felt required to outline them. These keywords define the scope at which variables or functions they encompass can be accessed from. The public section inside a class can be accessed from everywhere on or inside the object. Whereas the private variables can only be accessed from inside member functions (or friend functions). Protected members can be are similar but can also be accessed by derived classes via inheritance.


class A {

protected:
    int id;
};
 
// B inherits from A
class B : public A {
 
public:
    void setId(int val)
    {
 
        // Child class is able to access the inherited
        // protected data members of the base class
 
        id = val;
    }
 
    void displayId()
    {
        std::cout << "id is: "
             << id << std::endl;
    }
};

Note: difference for class vs struct, members in classes are Private by default and for structures Public.

Constexpr

  • its type must be a LiteralType
  • it must be immediately initialized
  • the full-expression of its initialization, including all implicit conversions, constructors calls, etc, must be a constant expression

The final point leads to:

Defines an expression that can be evaluated at compile time. Such expressions can be used as non-type template arguments, array sizes, and in other contexts that require constant expressions, e.g.

int n = 1;
std::array<int, n> a1;  // error: n is not a constant expression
const int cn = 2;
std::array<int, cn> a2; // OK: cn is a constant expression

Constexpr is a way of defining a constant something at compile time, function or variable. The rules between different languages specifications vary so its not easy to define exactly what can be constrexpr but if it can be easily defined at compile time than it might be a candidate for labeling as such, assuming that if it did call any other functions they also had that behaviour. In my personal opinion this is where knowing what your programming comes in, with high performance computing defining when things can be known at compile time may be very advantagous but in other scenarios where static analysers tell you a function can be constexpr but performance really isn’t any advantage it doesn’t offer any real benefit.

Mutable and Const

Personally I think const is a relatively simple concept to understand, something marked with const is constant, so can’t be modified. A function on a class therefore:

void Foo() const
{
  ...
}

tells the compiler I promise I won’t modify any of the internal state of the class. Mutable is something that allows that all to break and isn’t something again I knew about until I was scrolling around looking for keywords I didn’t know to much about…

The write up here is really good and the code I’ve listed below. A very similar but less well written example is given on the docs. The examples revolve around counters and where you would want to count how many times an object or member is accessed or maybe locked and therefore increment that member but the rest of the class is still constant. Thus you can allow the counter member to be mutable to achieve that purpose but still have the promise of constant-ness.

class MyClass
{
private:
    mutable int counter;
public:

    MyClass() : counter(0) {}

    void Foo()
    {
        counter++;
        std::cout << "Foo" << std::endl;    
    }

    void Foo() const
    {
        counter++;    // This works because counter is `mutable`
        std::cout << "Foo const" << std::endl;
    }

    int GetInvocations() const
    {
        return counter;
    }
};

int main(void)
{
    MyClass cc;
    const MyClass& ccc = cc;
    cc.Foo();
    ccc.Foo();
    std::cout << "Foo has been invoked " << ccc.GetInvocations() << " times" << std::endl;
}
