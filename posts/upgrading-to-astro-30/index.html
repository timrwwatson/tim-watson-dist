Upgrading to Astro 3.0

Recently I became aware (bombarded) by videos about Astro’s latest version. 3.0. There seems to be some really cool features in there, for those who want snazzy transitions or use those features. But for me I wasn’t THAT bothered by having or trying to add them. I’m quite content with my light/dark mode toggle. But it’s always good practice to stay up-to-date and the image optimisation seemed relatively interesting so why not go through the pain of upgrading my project from 2.3 to 3.0.

Steps to Upgrade

The steps are actually really easy, or atleast were for me:

npm install astro@latest
npm install @astrojs/react@latest @astrojs/tailwind@latest

By simply running these commands basically everything was done. npm did complain at me for vulnerabilities, but running a few audit commands resolved that issue. I could then run npm run start as per usual and the only difference here is Astro has changed the default local port to 4321.

Styling Changes

It seems that by updating react and tailwind the styling of the website has changed, most notably the size/space the blog posts take up, the navigation buttons in the header and image outline has reappeared!

The most obvious one is the header bar, as I spent alot of time aligning these buttons and they have now vertically aligned themselves again… A quick bit of debugging using inspect in the browser shows that they inherit a grid property from the nav ul class so simply removing this resolves the issue.

The border issue I believe was caused by running the npm run format linting/prettier commands which modified my base.css file:

prose-img:!mt-1 prose-img:border-2
prose-img:border-skin-line;

Specifically these lines at the bottom of .prose seem to have become un-commented along with other commented variables have moved. Again re-commenting them removes the orange borders from the images on the site.

Another thing that has ever so slightly changed are the page tags at the bottom of each page. These are used as links to filter all the pages with common tags, but it seems the # character styling has changed. Going into the styling for the Tag.astro property there isn’t any changed variables but can change some values to put some seperation in to make it slightly nicer to look at. Below we can see the befor and after code.

@apply -mr-5 h-6 w-6 scale-95 text-skin-base opacity-80 group-hover:fill-skin-accent;
@apply -mr-4 h-5 w-5 scale-75 text-skin-base opacity-70 group-hover:fill-skin-accent;

Finally the size of the site, expanding doesn’t seem obvious as to why it has happened. I think this is probably due to react upgrading but if anything it makes the site slight easier to read and because it has applied universally (as far as I can tell), I am going to leave it as it is.

MDX and Image tags

The one big feature I was expecting that might actually affect me was the <Image> tag and auto image optimisation. However what I didn’t really gleam from the youtube summaries I was bombarded with was that the new <Image> tag is available in .mdx and .astro files. And I primarilly write my blog posts in .md. So when I went to trial these features I realised that I needed to change a few different things. First of all I needed to install the dependancy for mdx to my astro project.

npx astro add mdx

npm install @astrojs/react@latest @astrojs/tailwind@latest

But once that issue was resolved and we have created the mdx file (this blog post), we then need to move/add some images to the src folder. Currently I have been locating my images in the public folder. Once included I can import them such as:

import { Image } from 'astro:assets';
import grid from '../../assets/images/grid.png';
import grid_site from '../../assets/images/grid_site.png';
import tags from '../../assets/images/tags.png';

and use them:

<Image src={tags} alt="The slightly modified page tags" title="The slightly modified page tags" />

Now that we are using the Astro Image tag, it converts/optimises the images. Such that they should now appear in webp format and when they are loaded in the browser will change depending on when they are visible on screen.

This all seems to work, but the amount compressed seems to range massively depending on if the image is already (somewhat) compressed, the size etc. If the image is already compressed it may not be worth going to this trouble and may already look worse… See below the two images from a lottery blog I have been writing, the first is using the <img> tag and if you inspect using your browser will appear as the same size as the second <Image> tag. However you will notice that because the images are hidden only one has appeared, until you open the New Image tag, at which point it will load.

Summary

In summary the upgrade wasn’t that difficult for me, but it also didn’t give me much. It’s always good to stay up to date and maybe I will challenge myself to add some transitions between pages. I had to do a little bit of maintance to the styling but nothing to major. And if I want to, I can upgrade my pages to mdx and get some optimisations but also it doesn’t seem to be massively different to what I currently have, which is also nice to know so I don’t need to waste time converting pages.
